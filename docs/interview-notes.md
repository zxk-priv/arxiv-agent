# 面试问答笔记

## 1. 这个项目解决什么问题

它把“最新 arXiv 论文推荐”拆成了两类入口：

- 用户明确输入关键词
- 系统从本地 PDF 论文库推断研究偏好

然后统一检索当天最新论文，并生成中文总结卡片。

## 2. 为什么要有 `paper_datasets`

因为很多时候用户自己都不容易用一句 query 准确描述研究偏好。

已有论文库其实就是一种更强的偏好信号：

- 看过什么
- 收集了什么
- 最近关注什么方法和任务

所以先把它转成结构化偏好，再去做推荐，通常比一次性手写 query 更稳。

## 3. 为什么不用全文 PDF

当前目标不是做精细问答，而是做研究偏好归纳。

标题页、摘要、引言前几页通常已经够用。只读前几页可以：

- 更快
- 更省 token
- 更少噪声

## 4. 为什么用 Markdown 缓存

因为项目既强调工程可运行，也强调面试可解释。

Markdown 的优点是：

- 人可以直接阅读
- 程序可以稳定解析
- 可以很方便展示给面试官看

## 5. 为什么 RAG 不做 chunking

因为当前检索目标是“论文级推荐”，不是“段落级问答”。

一天论文只有几百篇，每篇摘要长度也不长，所以用“单篇论文一个文档”更简单、也更贴近业务目标。

## 6. 为什么不是 BM25 + 向量检索

当前实现优先选择轻量方案：

- FAISS 向量召回
- 启发式关键词加分
- 混合排序

这样做更容易落地、维护和解释。如果后续规模增大，再引入 BM25 或 reranker 会更合理。

## 7. 如何减少重复模型调用

项目用了三层缓存：

- 当天论文缓存
- PDF 偏好画像缓存
- 最终推荐结果缓存

其中最关键的是第二层：PDF 偏好缓存。

它能保证大部分 PDF 只分析一次，后续只补新文件。
